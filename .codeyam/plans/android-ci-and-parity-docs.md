## Summary

Bring the two-platform repo to a finished, presentable parity state: add an
Android CI job (Gradle build + unit tests) alongside the existing Swift CI,
document Android setup for contributors, refresh the README to present the app
as cross-platform with both scenario galleries, and confirm the codeyam audit
gate is green across both apps. This is the closing plan of the Android-port
sequence.

## Key Decisions

- **Android gets its own CI job, parallel to Swift.** The existing `ci.yml`
  builds/tests the Swift package on macOS. Add a separate job (Ubuntu +
  JDK + Android SDK) that runs `./gradlew` build + unit tests for `android/`, so
  each platform's checks run independently and a break in one is legible.
- **Separate Android setup doc, mirroring `MOBILE_SETUP.md`.** iOS setup lives
  in `MOBILE_SETUP.md`; add `ANDROID_SETUP.md` (emulator/SDK prerequisites, the
  `start-simulator kotlin-android-compose` run path) rather than overloading the
  iOS doc.
- **README presents both platforms and both galleries.** Update the intro and
  build sections to say the counter now ships for iOS and Android, and add an
  Android scenario-gallery block beside the iOS one so the parity is visible at
  a glance.
- **Audit-green is the acceptance bar.** The plan is done when
  `codeyam-editor` reports both apps consistent — glossary/test-registry cover
  the Kotlin entities, scenarios are fresh, and no untested-entity regressions
  exist on either side.

## Implementation

### 1. Add the Android CI job

**File**: `.github/workflows/ci.yml`

Add a job (e.g. `android`) that: checks out, sets up JDK (per the scaffold's
Gradle config) and the Android SDK, then runs the Gradle build and unit tests in
`android/` (`./gradlew :app:assembleDebug :app:testDebugUnitTest` or the
scaffold's equivalent). Keep it independent of the Swift job. Ensure the CI
badge in the README still reflects the whole workflow.

### 2. Document Android setup

**New file**: `ANDROID_SETUP.md`

Mirror `MOBILE_SETUP.md`: required JDK/Android SDK + emulator image, a
pre-flight checklist, and the `codeyam-editor editor start-simulator
kotlin-android-compose` run command, plus the Gradle build/test commands.

**File**: `CONTRIBUTING.md`

Add an Android build/test section pointing at `ANDROID_SETUP.md`, matching the
existing iOS contribution instructions.

### 3. Refresh the README for cross-platform

**File**: `README.md`

- Intro/"Build and run locally": state the app ships for **iOS** (`ios/`) and
  **Android** (`android/`), with build commands for each.
- Add an **Android scenario gallery** section beside the iOS one, referencing
  the Android screenshots captured in the UI plan (under
  `.codeyam/scenarios/screenshots/…`). If the gallery is generated by a codeyam
  command, regenerate it rather than hand-listing.

### 4. Verify full-repo parity and audit

**Commands**:

- `codeyam-editor editor assess` — both apps' sources/tests are inventoried; no
  untested-entity regressions on Android.
- `codeyam-editor editor refresh-tests` — Swift and Kotlin suites both green.
- The codeyam audit gate passes across both `apps[]` entries (fresh scenarios,
  consistent glossary/registry).
- Both previews boot: `start-simulator swift-ios-swiftui` and
  `start-simulator kotlin-android-compose`.

## Reused existing code

- `.github/workflows/ci.yml` — extended with a parallel Android job (Swift job
  unchanged).
- `MOBILE_SETUP.md` — the shape/template `ANDROID_SETUP.md` mirrors.
- `CONTRIBUTING.md` / `README.md` — extended, not rewritten, to cover both
  platforms.
- The Android scenarios captured in the UI plan — surfaced in the README
  gallery; not re-captured here.
- **Config-field survey:** no new codeyam config field or gate dimension — this
  plan adds a CI job, docs, and runs existing audit/verify commands. The Gradle
  `staticChecks`/`testRunners` entries were added in the scaffold plan.

## Scenarios to Demonstrate

- CI: the Android job builds `android/` and runs its unit tests green in a fresh
  CI run, independently of the Swift job.
- README renders both scenario galleries (iOS and Android) side by side.
- `codeyam-editor editor assess` shows both apps with no untested-entity
  regressions and a green audit.
- Both simulators boot the respective apps to the same `counter-active-count`
  state (final visual parity check).
